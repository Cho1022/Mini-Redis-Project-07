# Mini Redis

이번 프로젝트의 목표는 단순히 key-value 저장소를 만드는 것이 아니라, **Redis의 핵심 개념인 해시 테이블 기반 저장, TTL, RESP 프로토콜, TCP 서버 흐름, 영속성 확장 가능성**까지 직접 구현하고 설명할 수 있는 수준까지 도달하는 것이었습니다.

우리는 프로젝트를 4개의 역할로 나누어 병렬로 개발했고, 각 모듈이 서로 연결되어 하나의 Mini Redis 서버로 동작하도록 구성했습니다.

---

# 1. Redis 사용 이유


- 초고속 처리 속도 (인메모리): 모든 데이터를 메모리에서 관리하여, 일반적인 디스크 기반 데이터베이스(MySQL, PostgreSQL 등)보다 수십 배 이상 빠른 읽기/쓰기 속도를 제공합니다.
- 캐싱 (Caching): 자주 조회되는 데이터를 레디스에 저장해 두어, 데이터베이스(DB)에 대한 부하를 줄이고 서비스 응답 속도를 높입니다.
- 다양한 데이터 구조 지원: 단순 키-값(Key-Value) 형태를 넘어 리스트, 세트, 해시, 정렬 세트(Sorted Sets) 등 다양한 데이터 구조를 지원하여 랭킹 보드 등 복잡한 실시간 기능 구현이 쉽습니다.
- 영속성 및 확장성: 메모리 기반임에도 데이터를 디스크에 저장(Persistence)할 수 있으며, 레디스 클러스터를 통해 대용량 트래픽에 대응하는 확장성을 제공합니다.

# 2. 프로젝트 목표

이번 과제의 핵심 목표는 다음과 같았습니다.

- 해시 테이블을 활용한 key-value 저장소 구현

# 4. 중점 포인트에 대한 답변

## 해시 테이블 설계 원리, 구현 과정
해시 테이블은 key를 넣으면 value를 빠르게 찾기 위한 자료구조입니다.
key를 기준으로 저장,
key를 기준으로 바로 조회,
평균적으로 저장/조회/삭제를 빠르게 한다의 흐름 위주로 설계하였습니다.
해시 충돌이 났을 때는 chaining 방식을 사용하였습니다.
chaining은 해시 충돌이 났을 때 한 칸에 하나만 두는 게 아니라, 같은 칸에 여러 값을 연결해서 저장하는 방식입니다.


## 동시성
멀티스레드 대신 단일 이벤트 루프 기반 구조를 사용했습니다. 이유는 같은 key를 여러 스레드가 동시에 수정할 때 race condition이 생길 수 있기 때문입니다. 멀티스레드는 처리량 장점이 있지만 lock이 필요해 구조가 복잡해지므로, 그래서 이번 Mini Redis는 복잡한 병렬성보다 단순하고 일관된 실행 흐름을 우선했습니다. 이를 통해 구현 난이도를 낮추고, 저장소 상태를 더 예측 가능하게 유지하려고 했습니다.

## 만료 처리
TTL은 ExpirationManager가 만료 시각을 따로 저장하고, EXISTS, TTL, DEL 등의 접근 시점에 lazy expiration으로 처리합니다. 즉 만료된 키는 백그라운드에서 미리 제거하는 것이 아니라, 접근할 때 확인 후 제거합니다.또한 값 저장소와 만료 시간 저장소를 분리함으로써, value 관리와 TTL 관리의 책임을 명확히 나눌 수 있었습니다.

## 외부 사용 구조
현재는 HTTP API 대신 TCP + RESP 인터페이스를 제공하여 외부 클라이언트가 Redis 스타일 명령을 사용할 수 있도록 했습니다. 즉 사용자는 단순 내부 클래스 호출이 아니라, 실제 서버에 접속해서 SET, GET, DEL, EXPIRE, TTL 같은 명령을 Redis와 유사한 방식으로 보낼 수 있습니다. 이를 통해 “외부에서 사용할 수 있는 저장소”라는 요구사항을 충족하였습니다.

## 무효화 방식
현재 invalidate는 delete와 동일하게 즉시 삭제하는 방식으로 구현했습니다. 즉 사용자가 특정 key를 무효화하면, 해당 값은 저장소와 TTL 관리 구조에서 함께 제거됩니다. 이 방식은 단순하고 동작이 명확하다는 장점이 있습니다. 다만 실제 서비스에서는 soft invalidation처럼 “삭제하지는 않지만 더 이상 유효하지 않다고 표시하는 방식”도 사용할 수 있기 때문에, 향후 deprecated 플래그 기반 방식으로 확장할 수 있도록 여지를 남겨두었습니다.

## 서버 다운 시 데이터 유지
기본 저장소는 메모리 기반이므로 서버 종료 시 데이터 유실 가능성이 있습니다. 이를 보완하기 위해 현재 상태를 저장하는 RDB snapshot 방식과, 쓰기 명령을 기록하는 AOF append/replay 방식을 설계하고 구현했습니다.  RDB는 특정 시점의 전체 데이터를 통째로 저장하는 방식이고, AOF는 SET, DEL, EXPIRE 같은 쓰기 명령을 순서대로 기록하는 방식입니다. 이를 통해 메모리 기반 저장소의 약점을 보완하려 했습니다.

# 5. AI 활용 회고

AI를 통해 프로젝트 초반에 전체 구조를 빠르게 잡을 수 있었고,  
Redis를 처음 접하는 상태에서도 TCP 서버, RESP, 해시 기반 저장소, TTL, persistence 같은 개념을 코드 구조로 옮기는 데 큰 도움을 받았습니다.  
특히 “어떤 파일을 어떤 책임으로 나눌지”, “누가 어떤 역할을 맡아야 하는지”, “어떤 흐름으로 서버가 동작해야 하는지”를 정리하는 과정에서 AI의 도움을 많이 받았습니다.

이번에 기장 크게 느꼈던 문제점은 할루시네이션 문제였습니다. RDB/AOF를 구현 완료하고 난 뒤에 콘솔에서 RDB/AOF반영이 되지 않아서 이유를 확인하고자 AI의 도움을 요청하였습니다.
하지만 AI는 자꾸 즉시 저장 기능이 문제라고 진단만 하는 문제가 있었습니다. 나중에 확인해보니 콘솔에 접속하는 코드에 명령인자에 포트가 적혀져 있지 않아서 RDB/AOF와 동기화 되지 않은 문제가 있었습니다.
엉뚱한 답변이라는 것을 눈치 채지 못하였다면 이상한 행동을 반복했을 것이지만 코드를 어느정도 이해하는 것으로 방지할 수 있었습니다.
그래서 AI가 만든 코드를 어느정도 이해를 하는 것이 필요로 합니다.

가장 대표적인 문제는 “파일은 생겼는데 실제로는 완전히 연결되지 않은 상태” 였습니다.
예를 들어 RDB/AOF는 구현됐지만 서버 시작/종료 흐름과 완전히 통합되지는 않았고,
