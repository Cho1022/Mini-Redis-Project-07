Mini Redis Integration Progress

목적
- 단위 테스트 이후 실제 요청/응답 흐름 검증 단계를 보강했다.
- 서버 시작 진입점, 실제 클라이언트 접속 경로, 통합 테스트를 명확히 정리했다.
- persistence 와 router 는 아직 기본 흐름 다음 단계라는 점을 분리해서 유지했다.

이번 단계에서 정리한 것
1. 서버 시작 진입점
- src/main.py 에서 python -m src.main 형태로 실행 가능하다.
- build_server() 로 조립 단계를 분리해 테스트와 실제 실행에서 같은 경로를 사용한다.

2. 실제 클라이언트 접속 경로
- scripts/demo_client.py 를 추가했다.
- RESP payload 를 직접 만들어 서버에 보내고 응답을 바로 확인할 수 있다.
- 예시:
  - 서버 실행: python -m src.main --host 127.0.0.1 --port 6379
  - 클라이언트 실행: python -m scripts.demo_client --host 127.0.0.1 --port 6379 "PING" "SET user:1 kim" "GET user:1"

3. 통합 테스트 보강
- tests/integration/test_tcp_server_flow.py 에서 단순 end-to-end 외에 두 가지 케이스를 추가했다.
- fragmented input
  - recv 가 한 번에 안 들어오는 상황에서도 parser 와 server buffer 로직이 정상 동작하는지 확인
- multiple commands in one buffer
  - 한 번의 read 안에 여러 RESP 명령이 들어와도 parse -> dispatch -> encode -> send 가 순서대로 처리되는지 확인

4. 현재 단계 해석
- 지금은 persistence / router 를 붙이는 단계가 아니라, 기본 흐름 완성도 보강 단계가 맞다.
- storage 단위 테스트가 통과해도 실제 socket 흐름에서는 buffer, partial read, protocol error 처리에서 자주 깨지므로 이 부분을 우선 검증했다.

수정 파일
- scripts/demo_client.py
- tests/integration/test_tcp_server_flow.py

현재 실행 방법
1. 서버 시작
python -m src.main --host 127.0.0.1 --port 6379

2. 클라이언트 접속 테스트
python -m scripts.demo_client --host 127.0.0.1 --port 6379 "PING" "SET user:1 kim" "GET user:1" "EXISTS user:1"

3. 테스트 실행
python -m pytest -q

이번 단계에서 확인된 것
- 서버 시작 진입점 동작
- 실제 TCP 연결 가능
- recv -> parse -> dispatch -> encode -> send 흐름 유지
- fragmented request 처리 가능
- multiple commands in one buffer 처리 가능
